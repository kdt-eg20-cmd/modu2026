# Job-MVP Feature Map (기능 우선순위 및 리스크 대응 전략)

> **문서분류:** MVP 기능 명세 및 개발 로드맵 우선순위
> **작성 목적:** 이전 단계의 JTBD(고객의 해결 과제)와 AOS-DOS 매트릭스 데이터를 실제 제품의 '개발 기능(Feature)'으로 매핑합니다. 단순 추정이 아닌, 국내 이커머스 API 정책 및 식약처(MFDS) 규제 등 **현실적인 제약과 데이터**를 기반으로 구현 난이도와 리스크 방어 전략을 설계했습니다.

---

| :--- | :--- | :---: | :---: | :---: | :---: |

## 2. 현실 데이터 기반 리스크(Risk) 및 대응 전략 (Mitigation)

초기 스타트업이 위 MVP 기능(F1~F4)을 구현할 때 직면하게 될 실제적인 기술적, 법적 장애물과 그 우회 전략입니다.

### 🔴 F1 리스크: 크롤링 봇(Bot) 차단 및 이커머스 약관 위반 문제
*   **현실 상황:** 쿠팡, 네이버 쇼핑 등 거대 이커머스 채널들은 무단 크롤링(웹 스크래핑)에 대해 강력한 Anti-Bot 솔루션(예: Cloudflare, Akamai)을 활성화해 두었으며, IP 차단 및 법적 분쟁 리스크가 매우 큽니다.
*   **대응 전략 (Mitigation):** 무식한 크롤링 대신 **공식 Affiliate(제휴) API**를 파싱하는 데이터 엔지니어링을 채택합니다. 쿠팡 파트너스 API, iHerb Affiliate Open API 등은 제휴 마케터에게 가격과 재고 메타데이터를 합법적이고 안정적으로 제공합니다. 해당 공식 API 텍스트에서 '총 중량(g)' 및 '정(캡슐) 수' 키워드만 정규 표현식으로 추출하는 방향으로 개발 난이도(서버 부하 리스크)를 대폭 낮춥니다.

### 🔴 F2 리스크: "건강기능식품에 관한 법률" 위반 (허위 과장 광고 심의)
*   **현실 상황:** 의사/약사가 아닌 플랫폼이 자체적으로 "이 성분은 암을 예방합니다" 혹은 "관절염에 100% 효과가 있습니다"라고 배지(Badge) 텍스트를 구성할 경우, 한국 식약처(MFDS)의 **질병의 예방 및 치료 표시·광고 금지 규정**에 즉각 위반되어 서비스 정지 처분을 받을 수 있습니다.
*   **대응 전략 (Mitigation):** 개발 난이도가 높은(법적 책임이 큰) 자의적 해석을 버립니다. 배지 데이터베이스는 **건강기능식품공전(식품의약품안전처 고시)**에 명시된 **'단어 본연의 기능성 인정 내용'**만 그대로 호출하여 래핑(Wrapping)합니다.
    *   *잘못된 예:* "치매 예방 효과" (X)
    *   *식약처 데이터 기반 정답 예:* "기억력 개선에 도움을 줄 수 있음 (식약처 인정 원료)" (O)
    *   이와 같이 판별식 배지를 철저히 '행정처 공식 DB 기준'으로 한정하여 스타트업의 법적 책임을 100% 방어합니다.

### 🔴 F3 리스크: K-Factor(바이럴 계수) 저하 및 카카오 링크 블락
*   **현실 상황:** 공유 메시지 클릭 후 '회원가입' 벽이 가로막고 있으면 전환율은 10% 미만으로 곤두박질칩니다.
*   **대응 전략 (Mitigation):** MVP 단계에서는 1-Tap 카드 수신자(예: 50대 부모님, 극단1 페르소나)에게 앱 설치를 절대 유도하지 않습니다. Open Graph(OG) 이미지가 들어간 웹 브라우저 링크(Web-view)로 구동되며, 카카오톡 내장 브라우저에서 '지금 바로 최저가 구매하기(쿠팡 앱 등 외부 연결)'로 다이렉트 랜딩 시켜 **구매 결정까지의 마찰을 0으로 만듭니다.**

### 🔴 F4 리스크: 원본 라벨 이미지 OCR 인식 오류
*   **현실 상황:** 해외 직구 영양제의 미세한 2pt짜리 스펙 텍스트를 무료 Tesseract OCR 모델로 읽어내려 하면 인식률이 50%를 밑돌아 잘못된 정보를 제공할 수 있습니다 (이는 극단2 페르소나가 극도로 혐오하는 상황).
*   **대응 전략 (Mitigation):** 초기 MVP 단계에서는 텍스트 변환을 억지로 시도하는 AI 비용을 제거하고, 단순히 **원본 제품 매뉴얼 이미지 원본(JPEG) 팝업 제공**만 지원합니다. 대신 "오류 라벨 신고 버튼"을 가장 크게 배치하여, 사용자가 크라우드소싱 형태로 오류 입력을 자발적으로 정정하게 하는 커뮤니티 데이터망을 활용합니다.
